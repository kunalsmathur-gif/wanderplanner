# WanderPlanner — Scaling & Technical Risk Assessment

**Basis for this document:** Full review of `docs/PRD.md`, `docs/system-design.md`, `docs/rag-strategy.md`, `docs/eval-set.md`, and the `apps/api` (FastAPI) + `apps/web` (Next.js) codebases as they exist today.

**Current architecture in one line:** Single-instance FastAPI service (Railway, free tier) + Qdrant (often `:memory:`) + Google Gemini + managed Postgres auth/analytics (Supabase) + cookie-based authentication, still with no queue and only limited in-process caching beyond that — deployed behind Vercel (frontend) and not yet designed for 1M MAU.

## 1. Security Vulnerabilities

> Historical findings snapshot from the pre-remediation review. See **§1a Remediation Status** for the current state after the auth/database/security rollout.

| # | Vulnerability | Evidence | Risk | Fix | When to fix |
|---|---|---|---|---|---|
| 1 | **Authentication/authorization was previously absent everywhere.** At the time of the finding, every route (`itinerary`, `chat`, `wizard-chat`, `compare-destinations`, `share`, etc.) was public. | `main.py` mounted all routers with no auth dependency; no `Depends()`/JWT/API-key check in any router. | Anyone could hit expensive LLM endpoints directly (cost abuse), scrape all traffic, or hit `/share/{slug}` at will. | Add API-key or session-token auth at the gateway/middleware level (even a lightweight signed cookie for anonymous sessions), rate-limit by identity, not just IP. | **Now, before any paid marketing push** — at the time of the finding, the API was a blank check to anyone with the URL. |
| 2 | **Unauthenticated, enumerable share links.** `routers/share.py` uses `uuid4().hex[:8]` (32 bits) in an in-memory dict with no ownership check. | `share.py:26-43` | Anyone can eventually enumerate/guess slugs (32-bit space is small at scale) and read other users' itineraries (PII: dates, destinations, group composition). Also resets on every deploy/restart. | Move to DB-backed store, use a 128-bit token (UUID4 full or `secrets.token_urlsafe(16)`), add optional expiry, and rate-limit `GET /share/{slug}` to blunt enumeration. | **Before 1,000 concurrent users** — low cost fix, high blast-radius risk. |
| 3 | **SSRF via `extract-trip` (Start Anywhere).** Endpoint accepts an arbitrary URL and server-side fetches it with redirects followed. | `routers/extract_trip.py`, `chains/extract_trip_chain.py:38-50` (httpx `follow_redirects=True`, only strips HTML tags) | Classic SSRF: an attacker can point this at internal Railway metadata endpoints, cloud metadata IP (`169.254.169.254`), or internal services, or use it to fetch/exfiltrate arbitrary web content through your server's IP. | Add an allowlist/denylist (block private IP ranges & metadata IP via `ipaddress` check on resolved DNS), cap response size, set `follow_redirects=False` or cap redirect hops, add a fetch timeout + content-type allowlist (text/html only). | **Now** — this is the single highest-severity finding. |
| 4 | **Prompt injection via free text + scraped content.** User trip descriptions, chat messages, and fetched web page text (via `extract-trip`) go straight into LLM prompts with no adversarial-input handling beyond a travel-topic guardrail prompt. | `chains/chat_chain.py`, `chains/extract_trip_chain.py`, `chains/feasibility_chain.py`, `chains/recommend_cities_chain.py`, `chains/chat_refine_chain.py` | A malicious page fetched via Start Anywhere, or a crafted chat message, can attempt to override the system prompt ("ignore previous instructions...") to leak the system prompt, produce off-brand/harmful content, or manipulate itinerary output (e.g., inject a phishing `booking_url`). | Add a prompt-injection classifier/guard step before including fetched or scraped text in prompts; sandbox retrieved content behind clear delimiters (already partially done) and instruct the model to treat it as data, never instructions; validate/allowlist `booking_url` domains before rendering as a clickable link in the frontend. | **Before scaling marketing/traffic** — moderate effort, meaningfully reduces brand/abuse risk. |
| 5 | **Verbose error responses leak internals.** Several endpoints return `str(exc)` directly in the HTTP 500 body. | `routers/chat.py`, `routers/feasibility.py`, `routers/recommend_cities.py`, `routers/wizard_chat.py` | Raw provider errors (e.g., Gemini quota messages, stack traces) can leak infra details, provider identity, and internal file paths to end users; useful recon info for attackers. | Wrap all router exception handlers in a generic error mapper: log full detail server-side, return a generic sanitized message + error code to the client. | **Now** — trivial fix, add to a shared FastAPI exception handler. |
| 6 | **No rate limiting anywhere** except a local Nominatim throttle. | `services/geocode.py:22-30` is the only throttle found; `/generate-itinerary`, `/chat`, `/wizard-chat`, `/compare-destinations` have none. | Any single client (or botnet) can drive unlimited LLM spend, since every `/generate-itinerary` call = several Gemini calls. This is a direct **financial DoS** vector, not just an availability one. | Add per-IP + per-session rate limiting (e.g., `slowapi`/Redis token bucket) on all LLM-backed endpoints; add a daily cost budget circuit-breaker per user/session. | **Now, before any public traffic spike** — this is the most financially dangerous gap. |
| 7 | **CORS with `allow_credentials=True` and env-configurable origins.** | `main.py:34-40`, `core/config.py:42-44` | If `ALLOWED_ORIGINS` is ever misconfigured (wildcard, staging domains left in prod), credentialed cross-origin requests become possible. Low risk today (no cookies/session used) but will matter once auth is added. | Keep an explicit, reviewed allowlist per environment; add a CI check that fails deploy if `*` appears in `ALLOWED_ORIGINS`; disable `allow_credentials` until real session cookies exist. | **When auth (fix #1) ships** — do them together. |
| 8 | **Sensitive-ish data in logs.** `print()` calls include destination names, trip details, and raw exception text. | `routers/travel_tips.py`, `chains/recommend_cities_chain.py`, `core/scheduler.py`, `services/comparison.py` | Not PII-severe today (no accounts), but as soon as user identity/email is added this becomes a compliance issue (GDPR/DPDP). Also, `print()` is not structured logging — unusable at scale for debugging/alerting. | Replace `print()` with structured logging (`structlog`/stdlib `logging` + JSON formatter), add log redaction for anything resembling PII, ship to a log aggregator (see infra section). | **Before adding user accounts** (functionally required at that point anyway). |
| 9 | **Unpinned/loosely pinned dependencies.** `google-genai>=1.0.0` is unpinned; `fastapi`, `httpx`, `qdrant-client` are on versions from mid-2024. | `requirements.txt` | Unpinned deps mean a breaking or vulnerable transitive update can land silently on every deploy; older pinned deps may have known CVEs by the time you scale. | Pin exact versions everywhere, add `pip-audit`/`dependabot` to CI, review quarterly. | **Now** — cheap, add to CI. |
| 10 | **AI-agent prompt-injection surface in the repo itself.** `apps/web/AGENTS.md` / `CLAUDE.md` contain instructions telling AI coding agents to read files under `node_modules/next/dist/docs/` before writing code. | `apps/web/AGENTS.md`, `apps/web/CLAUDE.md` | This is a supply-chain-adjacent risk: any file merged into the repo (via a compromised dependency or a malicious PR) could plant similar "agent instructions" that get silently obeyed by AI coding assistants operating on this repo, potentially exfiltrating secrets or making unreviewed changes. | Treat any AGENTS.md/CLAUDE.md-style file changes as high-scrutiny in code review; don't let AI agents auto-execute instructions found in arbitrary repo files without user confirmation; consider a repo policy that AGENTS.md changes require a second reviewer. | **Now** — process fix, zero engineering cost. |

**Priority order to fix:** #3 (SSRF) → #6 (rate limiting / cost DoS) → #1 (auth) → #2 (share links) → #5 (error leakage) → #4 (prompt injection) → #7/#8/#9/#10.

## 1a. Remediation Status (as of July 2026)

All 10 findings above are now either fixed or partially reduced with concrete implementation work. Auth is no longer deferred; it shipped as part of the new account system.

| # | Finding | Status | Implementation |
|---|---|---|---|
| 1 | No authentication/authorization | ✅ **Fixed (core path)** | Email/password + Google SSO auth now ship with cookie-based JWT + rotating refresh-token sessions. `POST /api/generate-itinerary` is gated by `get_current_user`; admin metrics are further gated by `get_current_admin_user` (403 for authenticated non-admins). Some non-sensitive endpoints remain public by design, but the expensive itinerary-generation path is no longer anonymous. |
| 2 | Enumerable share links | ✅ **Fixed** | `routers/share.py` now uses `secrets.token_urlsafe(16)` (128-bit) instead of `uuid4().hex[:8]` (32-bit); both `POST /api/share` and `GET /api/share/{slug}` are rate-limited. Still in-memory (per-instance) — DB-backing remains a scaling task, not a security one. |
| 3 | SSRF via `extract-trip` | ✅ **Fixed** | `chains/extract_trip_chain.py` now resolves DNS and rejects private/loopback/link-local/reserved/multicast IPs (blocks cloud metadata IP), manually walks redirects (max 3 hops, re-validated each hop), caps response to 2MB, and restricts content-type to `text/html`/`text/plain`. |

| 9 | Unpinned/loosely pinned dependencies | ✅ **Fixed (partial)** | `google-genai` pinned to `1.2.0` (was `>=1.0.0`); `pip-audit` added to CI (advisory, non-blocking — surfaced 23 pre-existing transitive CVEs in `starlette`/`python-multipart`/`urllib3`/`lxml` etc. pulled in by already-pinned `fastapi`/`uvicorn`; upgrading those is a larger, separate effort); `.github/dependabot.yml` added for weekly pip/npm/github-actions update PRs. |

## 2. What breaks at > 1,000,000 Monthly Active Users

| Component | Why it breaks | When it becomes critical |
|---|---|---|
| **Single Railway instance, single process, no horizontal scaling config** | `main.py`/`railway.toml` run one `uvicorn` process. There's no worker count, no autoscaling group, no load balancer config beyond Railway's default. One process cannot serve 1M MAU worth of concurrent LLM-bound + CPU-bound (embedding) requests. | Immediately at this scale — this is the #1 blocker. |
| **In-memory state everywhere (share links, geocode cache, travel-tips cache)** | `routers/share.py` uses a plain Python dict; multi-instance deployment means each instance has a different, inconsistent view of shared trips — users get random 404s depending on which instance they land on. Data is also lost on every restart/deploy. | As soon as you go beyond 1 instance (which you must, well before 1M MAU). |
| **Qdrant `:memory:` mode** | `.env.example` explicitly recommends `QDRANT_URL=:memory:` for "stateless Railway deployments." This means the entire RAG corpus (wiki/reddit/OSM/itinerary-cache vectors) lives in one process's RAM and disappears on restart, and cannot be shared across instances at all. | Must be fixed before *any* multi-instance deployment, i.e., well before 1M MAU — likely already needed at 1,000 concurrent users too. |
| **APScheduler running per-instance, uncoordinated** | Reddit refresh (6h) and OSM ingestion (weekly) are scheduled inside the same process as the API server. With N horizontally-scaled instances, you get N duplicate ingestion jobs hammering Reddit/Overpass simultaneously (and N-times cost/rate-limit violations against free public APIs like Overpass and Nominatim which explicitly rate-limit to 1 req/s). | As soon as instance count > 1. |
| **Dedicated database only partially covers product state** | Postgres now exists (Supabase in production) for accounts, consent, refresh tokens, password-reset tokens, and analytics events. However, itinerary history and share-link state are still not fully durably modeled there, and some user/session state remains browser- or process-scoped. | Continue expanding Postgres-backed persistence to itinerary/share/audit state before large-scale growth; the hardest part (introducing managed Postgres) is now resolved. |
| **Free-tier / single-region hosting (Railway free tier, Vercel)** | System design docs explicitly target "free tier" cost envelope (~₹15-30/mo at 100 users). Free tiers have hard CPU/RAM/bandwidth ceilings and no SLA; single-region hosting adds latency for users far from the region and has no failover. | Immediately — 1M MAU is 2-3 orders of magnitude beyond what free tiers are designed for. |
| **CPU-bound embedding model (`sentence-transformers`) running in the request path, in the same process as the API** | `core/embeddings.py` lazy-loads a `SentenceTransformer` and `CrossEncoder` in-process; every RAG retrieval + cache lookup + cache write calls `embed()` synchronously via `asyncio.to_thread`. This competes with the API's event loop's thread pool for CPU. | Becomes a bottleneck well before 1M MAU — this is actually one of the first things to break (see concurrency section below). |
| **Third-party free APIs (Nominatim, Overpass, Reddit public JSON, Open-Meteo)** | These are shared public services with explicit rate limits (Nominatim: 1 req/s; Overpass: best-effort, throttles/blocks abusers). At 1M MAU, geocode/best-time/travel-tips/OSM POI calls will vastly exceed these limits, resulting in the app being IP-banned from these free services entirely. | Immediately — these limits are already only "safe" at hundreds-of-users scale. |
| **LLM provider throughput/quotas** | Gemini API has per-project rate limits (RPM/TPM/RPD) tied to your billing tier. At 1M MAU with multiple itinerary generations per user, you will hit provider-side throttling regardless of your own infra, surfacing as the existing 429/503 retry path firing constantly (and each retry burns latency + still counts against quota). | Immediately at meaningful traffic — requires enterprise-tier LLM contracts, multi-provider routing, and possibly self-hosted open-weight models for cost/quota control. |
| **No CDN / caching for repeat itinerary requests at the edge** | Every itinerary request re-runs full retrieval + LLM generation (mitigated only by the semantic cache in Qdrant). No edge caching for popular destination combos. | Becomes a cost and latency issue well before 1M MAU. |

**Fix theme:** This is fundamentally a **re-architecture**, not a tuning exercise: move to stateless multi-instance API (behind a real load balancer/autoscaler), externalize all state (Postgres for accounts/itineraries/share-links, managed Qdrant Cloud or a horizontally-scalable vector DB, Redis for caches/rate-limits), move ingestion jobs to a separate worker/cron service decoupled from the API instances, and negotiate paid tiers with every third-party dependency (LLM, geocoding, weather).

**Right time to solve:** Start this the moment you commit to a growth roadmap targeting >50k MAU — these are 4-8 week engineering efforts each and should be sequenced *before* the traffic arrives, not reactively. Waiting until 1M MAU is imminent means doing all of it in a fire-drill.

## 3. What breaks at 1,000 Concurrent Users

This is a much nearer-term and more concrete risk than 1M MAU, since it's about instantaneous load on the current single-instance architecture.

| Component | Why it breaks at 1,000 concurrent | Fix |
|---|---|---|
| **Single uvicorn process/worker** | No `--workers` flag or process manager (gunicorn+uvicorn workers) configured in `Dockerfile`/`railway.toml`. Python's GIL + a single event loop means CPU-bound work (embedding, reranking) blocks other requests' async I/O scheduling on that same process. Rag-strategy doc's own load test shows throughput capping around **~23.6 req/s at concurrency=50** (and dropping to **~7 req/s** with reranking on) — nowhere near what 1,000 concurrent users need. | Run multiple uvicorn workers (`--workers N` sized to CPU cores) behind Railway, or move to gunicorn with uvicorn worker class; offload embedding/reranking to a separate process/service so the API event loop stays free for I/O. |
| **Embedding & reranking are CPU-bound and synchronous-under-the-hood** | `embed()` and `rerank_scores()` run PyTorch/sentence-transformers inference; even wrapped in `asyncio.to_thread`, this saturates the limited default thread pool and CPU on a Railway free/hobby-tier instance. | Move embeddings to a dedicated inference microservice (or a hosted embeddings API), autoscale it independently from the API; batch embedding calls where possible. |
| **In-memory share store / caches become a shared bottleneck & memory leak risk** | `_store: dict[str, dict]` in `share.py` and other in-process dict caches (geocode LRU, travel-tips 1h cache) grow unbounded under sustained load with no eviction beyond size-based LRU in one case — under 1,000 concurrent unique users, memory footprint climbs continuously until an OOM/restart, which also wipes all cached/shared data. | Move to Redis with TTLs for all these caches immediately; this is a one-sprint fix and gives you eviction, shared state across instances, and persistence across restarts for free. |
| **Qdrant `:memory:` under concurrent read/write** | Concurrent `search()` + `upsert()` calls (itinerary cache writes, Reddit/OSM ingestion) against an in-memory Qdrant instance from many simultaneous requests will contend for the same in-process lock/index, adding latency exactly when you need low latency. | Move to a real Qdrant deployment (self-hosted with persistent volume, or Qdrant Cloud) sized for concurrent read/write, sitting outside the API process. |
| **APScheduler ingestion jobs competing with user traffic for the same CPU/memory** | Weekly OSM ingestion iterates over dozens of destinations sequentially in the same process serving live traffic; if it happens to run during a traffic spike, ingestion + live requests fight for the same limited resources. | Move ingestion to a separate worker service/cron job (e.g., Railway cron service, or a separate container) so it never shares compute with the request-serving instances. |
| **No connection pooling limits on outbound httpx clients** | Each request may spin up its own volume of outbound calls (Gemini, Nominatim, Reddit, Overpass, Open-Meteo) without a shared connection pool cap; 1,000 concurrent users each triggering multiple outbound calls can exhaust file descriptors / OS-level connection limits on a small instance. | Use a shared, size-limited `httpx.AsyncClient` (connection pool limits) per external service, not one-off clients per call. |
| **No queue/backpressure for `/generate-itinerary`** | SSE endpoint runs LLM generation inline per request; with 1,000 concurrent generation requests, you'll either queue silently behind thread-pool/async limits (bad UX — timeouts) or the Gemini API itself starts 429-ing simultaneously, cascading retries (each up to 5 attempts × 3 models = up to 15 calls) that further amplify load. | Add a request queue with a concurrency cap sized to sustainable LLM throughput, return a "you're in a queue" status via the existing SSE status events rather than silently retrying 15x per request. |

**Right time to solve:** Most of these (workers, Redis for caches, Qdrant off `:memory:`, connection pooling) are **1-2 sprint fixes** and should be done **before your first real marketing push or viral moment** — they are cheap insurance relative to the cost of an outage during a traffic spike, and several of them (Qdrant persistence, Redis caches) are also correctness fixes, not just scale fixes.

## 4. API Cost Growth: 1M MAU × Multiple Itinerary Iterations

### Current documented cost baseline (from `rag-strategy.md` / `system-design.md`)
- ~100 active users/month ≈ **₹15-30/month** total (≈ ₹0.15-0.30/user).
- At 10,000 sessions/month, RAG optimization brought cost from **~$70/month → ~$30/month** (≈ **$0.003-0.007 per session**, down from ~7,500 raw tokens to ~600 tokens per generation thanks to context compression).
- These numbers assume **one itinerary generation per session**, using Gemini 2.5 Flash-Lite/Flash pricing, and **do not** account for regeneration/iteration behavior.

### Why cost grows faster than linearly with users at 1M MAU
1. **Iterations multiply per-user cost directly and non-linearly with engagement.** If the documented cost is ~$0.003-0.007 per itinerary generation, and each user regenerates/refines their itinerary an average of *k* times (wizard back-and-forth via `chat-refine`, "regenerate" after tweaking dates/budget/pace, comparison across 2-3 destinations each running its own generation), your effective cost per user is **k × base cost**, not base cost. At 1M MAU with even a conservative k=3-5 average iterations (wizard chat turns + itinerary regen + comparison + best-time + travel-tips + feasibility calls all being separate LLM calls per session), monthly cost scales as:
   - 1M users × 5 avg LLM-touching interactions × ~$0.005/call ≈ **~$25,000/month** just for the "cheap" cached-RAG path.
   - If cache hit rate for the itinerary semantic cache (0.88 cosine threshold) is low for a highly diverse 1M-user destination/date/pace matrix (very likely — the cache was designed and tuned for a small user base with concentrated demand), effective cost per generation reverts closer to the **pre-optimization ~$0.007-0.07/session** figures, pushing this well into **$50k-100k+/month** territory.
2. **Retry/fallback amplification.** `_gemini_itinerary()` retries up to **5 attempts × 3 models = 15 calls** on sustained transient errors (which become *more* likely at 1M MAU scale as you hit Gemini's own rate limits). Under provider throttling, a fraction of requests could each cost 3-15x the "happy path" token cost.
3. **Retrieval cost is not just LLM tokens.** Reranking (`cross-encoder/ms-marco-MiniLM-L-6-v2`) and embedding (`all-MiniLM-L6-v2`) run on your own compute per request — at 1M MAU this becomes a **compute cost line item** (CPU/GPU-hours), not just a per-token LLM API cost, and the rag-strategy doc's own numbers show reranking alone drops throughput from ~23.6 req/s to ~7 req/s per instance — meaning you need **~3.4x more compute capacity** just to keep reranking on at scale.
4. **Non-itinerary LLM-touching endpoints add up.** `chat`, `wizard-chat`, `chat-refine`, `feasibility-check`, `recommend-cities`, `extract-trip` are all separate Gemini calls; a fully-engaged user session could easily rack up 8-12 LLM calls before ever seeing a final itinerary.

### How to solve it
- **Cache-first, aggressively.** Raise itinerary-cache hit rate by broadening the cache key (already destination/duration/pace/purpose) with smarter approximate matching, and pre-warm the cache for top N destination/date-range combos.
- **Tiered model routing by cost-sensitivity.** Use a cheaper/smaller model (or self-hosted open-weight model) for low-stakes calls (chat guardrail replies, chip suggestions) and reserve the best model only for final itinerary generation.
- **Cap iterations per session with clear UX** (e.g., "3 free regenerations, then a lightweight edit mode that patches JSON without a full LLM call" — `chat-refine`'s patch-only path already exists per system-design, lean into it more).
- **Batch/precompute popular content** (best-time, travel-tips, recommend-cities for top destinations) instead of live LLM calls per user.
- **Real-time cost circuit breaker**: track $ spend per hour/day against a budget; degrade gracefully to RAG-skeleton/mock itineraries (Tier 2/3 fallback, which already exist!) when budget thresholds are hit, rather than an unbounded bill.
- **Negotiate committed-use/enterprise LLM pricing** once volume is predictable — spot/PAYG pricing is the most expensive way to serve 1M MAU.

**Right time to solve:** Cost modeling and the cache-hit-rate/iteration-cap work should happen **as soon as you have real usage data past 10k MAU** — that's when you can validate whether the current cache/compression assumptions (tuned on a much smaller, less diverse user base) actually hold. Doing this analysis *before* committing ad spend to acquire 1M users is essential — otherwise growth could be cost-negative.

## 5. Infra Limitations at Scale

| Limitation | Detail | Fix | When |
|---|---|---|---|
| **Free-tier Railway + single instance** | No autoscaling, no multi-region, resource caps typical of free/hobby tiers; documented cost model explicitly targets free tier. | Move to a paid, autoscaling compute platform (Railway paid plans, or migrate to ECS/Cloud Run/Kubernetes) with horizontal autoscaling rules tied to CPU/RPS. | Before 1,000 concurrent users. |
| **No load balancer / multi-instance readiness** | App has in-memory state (share store, caches) that breaks the moment you run >1 instance. | Externalize state (Redis, Postgres, managed Qdrant) *before* turning on multi-instance/autoscaling — otherwise autoscaling actively makes correctness worse (see share-link 404s). | Before enabling autoscaling — same milestone as above. |
| **Qdrant `:memory:` has no persistence, no clustering, no backup** | Explicitly documented as "data lost on restart"; not viable beyond a single instance or any durability requirement. | Move to Qdrant Cloud (managed, clustered) or self-hosted Qdrant with persistent volumes + backups + replica set. | Before 1,000 concurrent users / any multi-instance deployment. |
| **Database foundation exists, but persistence is incomplete** | Managed Postgres now exists for accounts, consent, password resets, refresh tokens, and analytics events. Remaining gap: itinerary history and shared-trip durability still lean on browser/in-memory patterns. | Extend the current Postgres footprint to durable itinerary/share/audit state rather than introducing an entirely new database layer. | Before scaling multi-instance user history and share-link reliability expectations. |
| **No CDN/edge caching** | Vercel gives some edge caching for static frontend assets, but API responses (best-time, travel-tips, geocode) aren't cached at the edge. | Add an API-level cache (Redis or a CDN like Cloudflare in front of the API) for cacheable, non-personalized endpoints (best-time, travel-tips, recommend-cities for common inputs). | Before 1M MAU; helps meaningfully even at 1,000 concurrent users. |
| **No observability stack** | Logging is `print()`-based; no APM, no error tracking (Sentry), no metrics/dashboards, no alerting mentioned anywhere in code or docs. | Add structured logging + an APM/error-tracking tool (Sentry/Datadog) + basic dashboards (request rate, latency p50/p95/p99, LLM cost per hour, cache hit rate, error rate) + alerting on cost/error spikes. | **Immediately** — you cannot safely operate at 1,000 concurrent users, let alone 1M MAU, without this; this is also a prerequisite for diagnosing every other issue in this document. |
| **No CI/CD gating for security/dependency risk** | `.github/workflows/ci.yml` exists but no evidence of dependency scanning, SAST, or secret scanning integrated. | Add `pip-audit`/`npm audit`/Dependabot, secret-scanning (e.g., gitleaks), and basic SAST to the existing CI pipeline. | Now — cheap, should ship alongside the security fixes in section 1. |
| **Single LLM provider dependency (Gemini)** | No provider-abstraction/fallback beyond model-tier fallback within Gemini itself; a Gemini outage or account-level quota issue takes down itinerary generation entirely (mitigated partially by cache/RAG-skeleton/mock fallback tiers, which is good design). | Add a secondary LLM provider (e.g., OpenAI/Anthropic/Groq — Groq is already partially wired) behind the same interface for true provider-level failover, not just model-tier failover within one provider. | Before 1M MAU — provider outages become a "when" not "if" at that volume/visibility. |
| **Third-party free API dependency (Nominatim, Overpass, Reddit)** | These have explicit low rate limits and no SLA; app could get IP-banned. | Move to paid tiers (e.g., Nominatim → Mapbox/Google geocoding, Overpass → a paid OSM POI provider) once volume exceeds their fair-use thresholds. | Before 1,000 concurrent users for Nominatim (1 req/s limit is trivially exceeded); before 1M MAU for the rest. |

**Right time to solve:** Observability + externalizing state are the two items with zero downside to doing early — start now regardless of current traffic, since they're also correctness/debuggability wins. Everything else should be sequenced against concrete growth milestones (1,000 concurrent → externalize state & Qdrant persistence; 10k MAU → cost/cache validation; 100k+ MAU → multi-provider LLM, paid third-party APIs, real autoscaling & multi-region).

## 6. Memory & RAG at Scale

### Is "memory" being used today?
Yes, in two distinct senses, both currently lightweight and both will surface problems at scale:

1. **Conversation/session memory (short-term, in the request path):**
   - `chat_chain.py` passes the **last 10 turns** of chat history into every Gemini call (no summarization/truncation strategy beyond a hard slice).
   - Wizard/trip-config state lives entirely in **frontend Zustand stores** (`wizardChatStore`, `tripConfigStore`, `chatStore`, etc.) — there is **no server-side session memory or persistence**; state is lost on tab close/refresh (explicitly documented as a known limitation in the PRD).
   - This means "memory" today is really just **client-side state + a sliding window of recent messages re-sent on every call** — there's no vector-based long-term memory of a given user's preferences across sessions.

2. **Corpus/knowledge memory (RAG, long-term, shared across all users):**
   - Qdrant collections (`wiki`, `reddit`, `osm_pois`, `itinerary_cache`) are the system's actual long-term "memory" — ingested once, refreshed periodically (Reddit every 6h, OSM weekly), and reused across all users' retrieval calls.

### How each surfaces problems at scale

**Conversation memory issues:**
- **Growing token cost per chat turn.** Sending the last 10 raw messages (uncompressed) on every call means average context size grows with more engaged/longer conversations; at 1M MAU with multi-turn wizard/chat flows, this is a meaningful and *linearly growing* cost driver that isn't captured in the current per-session cost model (which appears tuned around single-shot itinerary generation cost, not sustained chat cost).
- **No cross-session personalization possible.** Because there's no persisted user memory, you can't do the "generated-itineraries learning flywheel" the system-design doc calls out as future roadmap work — at 1M MAU this becomes a real product gap (competitors with persisted preference memory will feel more personalized).
- **Client-only state = no server-side recovery, no multi-device continuity, no abuse/audit trail.** At scale, users complaining about "lost my itinerary after refresh" support tickets will grow proportionally with MAU, with no server-side fallback.

**RAG/corpus memory issues:**
- **In-memory Qdrant is a single point of failure and doesn't scale horizontally** (already covered in sections 2/3/5) — this is the most urgent RAG-related infra risk.
- **Ingestion sources are too sparse for 1M MAU's destination diversity.** `KNOWN_DESTINATIONS` is a curated ~100-city list; Reddit/OSM ingestion only covers these. At 1M MAU, users will request itineraries for long-tail destinations with **zero ingested corpus content**, causing the RAG retrieval to return nothing (`"No pre-fetched research available"`) far more often — degrading itinerary quality exactly for the users generating the most incremental growth (long-tail destinations), and reducing the itinerary-cache hit rate benefit the cost model depends on.
- **Cache staleness & pollution risk grows with volume.** `itinerary_cache` is a shared collection across all users, written to by every successful generation; at high volume, low-quality or edge-case generations (e.g., unusual persona combos) could pollute the cache with a mediocre match that gets served to many unrelated users above the 0.88 cosine threshold if their query happens to land nearby in embedding space. There's no cache quality/feedback signal (thumbs up/down) feeding back into what stays cached.
- **Reddit/OSM ingestion running in the same process as the API** (already flagged in section 3) becomes a memory-contention and correctness issue at scale: as the corpus grows toward covering 1M-MAU-level destination diversity, ingestion jobs get bigger and longer-running, colliding more often with live traffic on the same instance.
- **No corpus size ceiling planning.** Qdrant `:memory:` mode will simply run out of RAM as wiki/reddit/OSM/itinerary-cache collections grow to cover many more destinations and accumulate cache entries from 1M users — there's no documented plan for corpus size vs. instance RAM.

### How to solve it
- **Short-term chat memory:** Replace the flat "last 10 messages" slice with a proper summarization strategy (rolling summary + last N turns) to keep token cost roughly flat regardless of conversation length; this is directly analogous to the context-compression work already done for RAG (time-decay + dedup + truncation) — apply the same discipline to chat history.
- **Server-side session persistence:** Add lightweight server-side session storage (Redis, keyed by an anonymous session ID/cookie) so wizard/chat state survives refresh — a stepping stone before full user accounts.
- **User accounts + long-term preference memory:** Once accounts exist, persist per-user preference embeddings/history to enable the "learning flywheel" already on the roadmap — this is a genuine product differentiator at scale, not just a scaling fix.
- **Externalize + scale the vector DB:** Move off `:memory:` Qdrant to a managed/clustered deployment with disk-backed persistence and monitoring on collection size/RAM.
- **Expand and prioritize ingestion by demand.** Instrument which destinations get requested but have no/thin corpus coverage, and prioritize ingestion expansion (more subreddits, more Wikivoyage pages, on-demand OSM ingestion for never-seen destinations) based on real demand signal rather than a static curated list.
- **Add a cache quality feedback loop.** Let a thumbs-up/down (or implicit signal like "user kept it without regenerating") gate what gets written/kept in `itinerary_cache`, to avoid quality regression as cache volume grows.
- **Decouple ingestion into its own worker/service**, independent of API instance count, with its own scaling and scheduling (fixes both the memory-contention and the "N instances = N duplicate ingestion jobs" problem from section 2/3 in one move).

**Right time to solve:** The RAG corpus/`:memory:`-Qdrant fixes are needed at the same time as the "1,000 concurrent users" infra fixes (section 3) — they're really the same underlying problem. The chat-memory summarization and cache-quality-feedback work should happen once you have real conversation-length and cache-hit-rate telemetry (i.e., right after shipping observability, section 5) — you need data to know whether these are theoretical or already-live cost/quality problems. Full user-account-based long-term memory is a bigger product+infra investment best timed with a broader "accounts" milestone, ideally planned well before 1M MAU rather than bolted on reactively.

## 6a. RAG Failure Modes at Scale — Where It Shines and Where It Breaks

RAG (retrieval-augmented generation, `docs/rag-strategy.md`) is what keeps itinerary generation grounded in real destination content instead of the LLM's parametric memory. It's genuinely earning its place today, but every strength has a corresponding scale-driven failure mode worth naming explicitly for demo/investor Q&A.

**Where it shines (today, at current scale):**
- Grounds itinerary generation in real, ingested Wikivoyage/Reddit/OSM content instead of LLM hallucination — directly measurable via the golden-dataset IR-metric eval (`eval/run_rag_eval.py`: Precision@10/Recall@10/MRR/nDCG).
- The 3-tier fallback chain (cache → RAG-skeleton → enhanced mock, §4 of `rag-strategy.md`) means an LLM outage or retrieval failure degrades quality gracefully instead of hard-failing the request.
- The eval harness already caught a real, live production bug — RAG silently returning nothing for months because of a missing Qdrant payload index — which is the strongest evidence this approach has real operational value, not just theoretical appeal.

**Where it starts failing as usage/geography scales:**
- **Long-tail destination coverage collapses first.** The curated corpus covers ~134 destinations (only 11 India-specific, despite India being the primary cohort — §8a). At meaningfully higher MAU with a longer geographic tail, retrieval returns `"No pre-fetched research available"` far more often for exactly the users driving incremental growth, and the model silently reverts to un-grounded general knowledge — the RAG safety net disappears precisely where new-destination demand is highest.
- **Storage ceiling is real, not theoretical.** The free 1GB Qdrant Cloud cluster comfortably covers today's ~134-destination corpus (~500K–800K vectors) many times over but is explicitly not sized for eager global ingestion (§8). Scaling coverage means either a paid tier or smarter demand-driven ingestion — not "just add more."
- **Ingestion pipeline fragility compounds at scale.** Reddit ingestion is currently 403'd in production (approval pending) — a live, present-day gap in the "hidden gems" signal, not a hypothetical one. More ingestion sources (YouTube, blogs, OSM) means more independent failure points, each silently degrading a slice of retrieval quality without surfacing as a user-facing error (the same failure-swallowing behavior that hid the payload-index bug for months).
- **Freshness decay under-refreshed at volume.** Time-decay scoring (18-month half-life) deprioritizes stale content; if ingestion refresh cadence doesn't scale with corpus size and destination count, a growing fraction of the corpus silently ages out of relevance without being replaced.
- **Cache pollution risk grows with volume** (already covered in §6 above) — low-quality generations written into `itinerary_cache` can get served to unrelated users above the similarity threshold, and there's no quality feedback loop yet gating what stays cached.
- **Reranking's throughput cost gets worse, not better, at scale.** Cross-encoder reranking causes a measured ~3x throughput drop under load — already forcing it to be scoped to a single call site rather than applied everywhere; at higher concurrency this tradeoff sharpens further and may need a dedicated, independently-scaled reranking service (already flagged as a scale-time item in §6).
- **RAG only reduces hallucination, it doesn't guarantee truth.** As more sources are ingested at scale (more Reddit posts, more blogs), the probability of ingesting stale, wrong, or low-quality source content rises — RAG will confidently retrieve and repeat bad source material with the same authority as good material, unless a source-quality/authenticity weighting system (already scoped in §8a) is built out ahead of ingestion-source growth, not after.

**Sequencing implication:** RAG's failure modes scale in lockstep with the same demand-driven-ingestion and vector-DB-hardening work already sequenced in §6 and §8 — there's no separate "RAG scaling" workstream to plan, but the corpus-coverage and source-diversification gaps (§8a) should be treated as directly limiting how far RAG's current benefits actually extend today, not just a future nice-to-have.

## 8. Global Destination Coverage — What Breaks Trying to Pre-Ingest "All Countries/Cities/Locations"

A natural-sounding growth plan — "ingest every country, state, city, and POI on Earth upfront" — looks like it just needs a bigger Qdrant tier. It doesn't. It breaks on three independent axes at once, and the fix is an architecture change (demand-driven ingestion), not a bigger free-tier plan.

### The math, grounded in this repo's actual chunking

Current baseline: `KNOWN_DESTINATIONS` (`apps/api/scrapers/reddit.py`) is a curated **134-destination** list. Wiki chunks are ~400–600 chars, Reddit chunks are per-paragraph, OSM POIs are capped at ~300/destination (`services/gems.py`'s bound). That's an estimated **~500K–800K vectors total** across the `wiki`/`reddit`/`osm_pois`/`itinerary_cache` collections today — comfortably inside Qdrant Cloud's 1GB free tier.

"All 195 countries → ~3,000–5,000 states/provinces → ~150,000–200,000 cities (standard population threshold) → POIs in each" is roughly a **1,000x increase in destination count** over today's curated list. Holding the same per-destination chunk volume constant, that's **~500M–800M vectors**, which means:

| Constraint | Today (134 destinations) | At ~150K destinations |
|---|---|---|
| Storage (vectors + HNSW index + payload, ~2–4KB/point) | ~1–3GB | **~1–3 TB** |
| Qdrant Cloud free tier (1GB) | Fits | Breaks by ~1,000–3,000x |
| OSM ingestion time (`osm_ingest_delay_seconds`, serial, polite to the free Overpass API) | 134 × 2s ≈ 4.5 min | 150,000 × 2s ≈ **83+ hours continuous — Overpass will very likely rate-limit/IP-ban long before this completes** |
| Embedding + corpus-extraction compute (Gemini calls, `all-MiniLM-L6-v2` batches) | Minutes, near-free | Real, sustained $ and hours — comparable to embedding a large slice of Wikipedia |
| Qdrant Cloud monthly cost at that storage tier | $0 | Likely **$1,000s–$10,000s/month** — this is enterprise-tier pricing, not "the next plan up" |

**This isn't a "which vector DB" problem — eager global pre-ingestion doesn't scale on any provider**, self-hosted or managed, because the underlying corpus size and ingestion-source rate limits are the bottleneck, not Qdrant specifically.

### The fix: demand-driven ingestion, not an eager global crawl

No real travel-RAG product eagerly ingests every populated place on Earth upfront — the overwhelming majority would never be requested, and you'd be paying to store and periodically refresh dead weight. The corpus should grow with **actual user demand**, converging on a long tail of a few thousand real destinations, not hundreds of thousands of speculative ones.

Concretely, against this repo's existing structure:

1. **Track ingestion state durably in Postgres** (the existing durable-state store — avoid adding a new DB layer per the principle in section 5 above):
   ```sql
   CREATE TABLE destination_ingestion_state (
       destination TEXT PRIMARY KEY,
       osm_last_ingested_at TIMESTAMPTZ,
       wiki_last_ingested_at TIMESTAMPTZ,
       request_count INTEGER DEFAULT 0,
       last_requested_at TIMESTAMPTZ
   );
   ```
2. **Add an `ensure_destination_ingested(destination)` gatekeeper**, called wherever `retrieve_context()` / `get_gem_intel()` currently fire in the itinerary-generation path. On a first-ever request for a destination: geocode-validate via Nominatim (rejects junk/misspelled input before spending Overpass/embedding budget), then run `ingest_osm_pois()` (`scrapers/osm.py`, already idempotent and per-destination) and `ingest_wikivoyage()` (`scrapers/wikivoyage.py` — already written per-destination, but currently **not wired into any calling path**, scheduled or on-demand) inline, accepting one-time added latency for that first user, then write the timestamp row. Reuse the same per-destination `asyncio.Lock` stampede-guard pattern already used in `services/gems.py` so a demand spike on one new destination doesn't trigger duplicate concurrent ingestion.
3. **Stop looping the scheduler over a static list.** `core/scheduler.py::_refresh_osm_pois` currently iterates the fixed `KNOWN_DESTINATIONS` on a timer; it should instead refresh only rows in `destination_ingestion_state` past their staleness window (`WHERE osm_last_ingested_at < now() - interval`) — i.e. refresh what's actually been requested, not the entire static list (and never the entire globe).
4. **Widen Reddit's destination matching.** `scrapers/reddit.py::_extract_destination()` currently only recognizes names already in the static `KNOWN_DESTINATIONS` list, so organically-mentioned destinations outside that curated set are silently dropped even as OSM/wiki coverage grows on demand elsewhere — it should match against `destination_ingestion_state` (or geocode-normalized extraction) instead.
5. **Guard against cost/abuse.** Rate-limit new-destination cold starts per IP/session (e.g., max 5/hour) so garbage input can't run up Overpass/Gemini spend. `request_count`/`last_requested_at` also gives a natural demand signal for later prioritizing "popular but currently thin" destinations for deeper ingestion, without ever having to eagerly crawl the long tail nobody asks for.

**Net effect:** corpus size tracks real usage, not the size of the world. Even at 1M MAU, realistically you converge on a few thousand actively-requested destinations — staying inside Qdrant Cloud's free-to-low-paid tiers indefinitely, rather than needing a multi-TB self-hosted cluster to cover places no user ever visits.

**Right time to solve:** this is the same milestone as the "expand and prioritize ingestion by demand" bullet already in section 6 above — do it once you have real telemetry on which destinations are actually being requested (post-observability rollout), and *before* anyone attempts a "let's just ingest everywhere" expansion. It's substantially cheaper to build demand-driven ingestion from the start than to walk back a global pre-crawl later.

## 8a. Hidden-Gem Source Diversification & India-Specific Coverage Policy (2026-07-16 evaluation)

Full research and roadmap sequencing lives in `docs/NEXT_SESSION_TODO.md`; this section records the architectural reasoning so it isn't re-derived from scratch.

**Problem:** `services/gems.py` derives 100% of its community sentiment signal from the single `reddit` Qdrant collection. Reddit ingestion is confirmed broken in production (403s since the Cloud migration) with no ETA on formal API approval. Without a second source, this differentiator degrades to an honest-but-empty result indefinitely if approval never arrives.

**Important distinction — this is separate from §8's demand-driven ingestion.** §8 covers OSM POI + Wikivoyage content, which is *already* demand-driven via `ensure_destination_ingested()` — no popularity cap exists there, any requested destination gets ingested regardless of India/global split. The coverage gap addressed here is in the three **curated, popularity-capped seed/tagging lists** that are a separate mechanism from demand-driven ingestion:

1. `KNOWN_DESTINATIONS` (`scrapers/reddit.py`, 134 entries total, only 11 India-specific) — used to tag scraped Reddit posts by destination for gem-sentiment mining.
2. `WIKIVOYAGE_ITINERARY_TITLES` (`scrapers/itinerary_corpus.py`, 5 entries, only 1 India-specific).
3. The planned YouTube video-discovery destination/query seed list (not yet built).

**Asymmetric India-coverage policy:** every curated/seed list above keeps the rest-of-the-world set capped by global popularity as today, but carries a deliberately larger India-specific subset, since India is the primary user cohort and domestic tier-2/3 destinations (hill stations, pilgrimage towns, heritage towns) are exactly this feature's target market. This is a list-expansion exercise, not a new architecture — cheap to do, asymmetric by design. Two concrete, zero-cost gaps found while investigating this: `scrapers/reddit.py`'s `SUBREDDITS` list (used for gem-sentiment mining) has no India subreddit at all, while `scrapers/itinerary_corpus.py`'s separate `ITINERARY_SUBREDDITS` list already includes `"IndiaTravel"` — the two lists should be reconciled. The sentiment lexicon (`_POSITIVE_WORDS`/`_NEGATIVE_WORDS`) is also English-only today, silently under-weighting Hindi/Hinglish commentary common in domestic-travel YouTube comments.

**Sequencing caution:** expanding the India destination set increases how many *new* destinations hit first-request OSM ingestion (`ensure_destination_ingested`) as real traffic arrives, funneling through the same Overpass rate-limiting problem already tracked in the `osm-poi-refresh-big-cities` todo (Mumbai/Delhi already confirmed problematic). Sequence the India list expansion after or alongside that fix, not before.

**Source evaluation summary (full table in `docs/NEXT_SESSION_TODO.md`):**
- **Build now, free:** YouTube transcripts (already partially wired), YouTube Data API v3 comment mining (free 10k-units/day quota — `search.list` 100 units/query, `commentThreads.list` 1 unit/call), expanded travel-blog RSS feeds.
- **Roadmap, paid, similar cost tier:** Google Places API (Text Search Pro $32/1,000, Place Details Pro $17/1,000, first 5,000/mo free — est. $170-320 one-time for a full destination-set refresh, then infrequent); TripAdvisor Content API (confirmed self-serve on sign-up, exact per-call rate not yet priced out — includes real review text, which Google's cheaper tiers omit).
- **Roadmap, lower priority:** X/Twitter API v2 (flat $200-5,000/mo subscription, noisy firehose — last-published pricing, unverified), Foursquare (public consumer tips/ratings API deprecated, now enterprise-sales-gated), Yelp Fusion (free but weak coverage outside US/UK/Canada), Pinterest (free but weak signal fit), Atlas Obscura (best editorial fit, no public API, needs a ToS check).
- **India-specific roadmap:** LBB (Little Black Book, no public API, ToS check needed), Thrillophilia (fold into existing affiliate-outreach list, same partner-gated shape as Viator/GetYourGuide).
- **Rejected — no viable access path:** Instagram (ToS/legal risk, Meta actively pursues scrapers), TikTok (Research API is academic-only, not for commercial use), Zomato (API closed to new developers since ~2019), Swiggy (no public API), MakeMyTrip/Yatra/Goibibo/Cleartrip (no self-serve developer APIs), Quora (no content API, ToS-prohibited scraping).

**Review/mention authenticity — no source hands us a single clean "reviewer trust" field.** Even Google's official Places API (New) Review resource has no reviewer account-age/tenure/Local-Guide-level field — only `authorAttribution.displayName/.uri/.photoUri`, `rating`, `text`, `publishTime`. Replicating the "click into a reviewer's profile" trick a human does on Google Maps would require scraping individual profile pages, which is out of scope (same ToS-risk category as the rejected sources above). The design instead builds a **composite, cross-platform authenticity weight** from signals each source's official API genuinely exposes, applied as a 0-1 multiplier on each mention's contribution to `mentions`/`pos_total`/`neg_total` (never a binary filter — the existing honesty guarantee that zero real mentions never produces a recommendation is preserved):

| Source | Signal | Cost |
|---|---|---|
| Reddit | Author account age, karma (via `/user/{username}/about.json`, public, unauthenticated) | Free |
| Reddit | Post score (already captured), temporal clustering, duplicate-text similarity | Free (derived from existing data) |
| YouTube | Commenter channel age, subscriber count, video count (via batched `channels.list`, 1 unit/call) | Free within quota |
| YouTube | Comment `likeCount` (already returned by `commentThreads.list`) | Free |
| Google Places (future) | Rating-velocity anomaly + rating-distribution shape, derived from `publishTime`/`rating` already being pulled | Free (no extra calls) |
| Google Places (future) | Reviewer account age / review-history depth | **Not available via official API** — out of scope, documented so it isn't re-explored |

**Rollout gate:** per explicit direction, none of the above build-now/roadmap-prep engineering work starts until this plan is committed to `docs/NEXT_SESSION_TODO.md` and both `main` and `feat/frontend-scaffold` carry the commit.

## 9. Single-Agent vs. Multi-Agent — When (If Ever) to Reconsider

WanderPlanner today is a **single-agent, multi-chain architecture**: one LLM (Gemini) invoked through 8 independently-prompted chains dispatched by deterministic backend routing — not an autonomous multi-agent framework (full classification in `docs/system-design.md` §1A and `TECHNICAL_DOCUMENTATION.md` §8B). This section exists to give explicit, concrete trigger conditions for when that assessment should be revisited, rather than leaving "should we go multi-agent" as an open judgment call at every scale milestone.

**Why single-agent holds at every scale tier currently roadmapped (up to 1M MAU, §2):**
- None of the scaling bottlenecks identified in this document (Qdrant `:memory:`, in-process ingestion, auth/session persistence, cost growth) are solved or made worse by introducing multiple interacting agents — they're infra/data problems, not orchestration problems.
- Multi-agent patterns (planner→critic→executor loops, tool-calling round-trips between agents) directly worsen two of this document's named risks: **API cost growth at scale** (§4 — each agent hop is a separately billed LLM call) and **latency** (the PRD's 15–20s generation budget is already tight with a single model call chain).
- The eval infrastructure (`apps/api/eval/`) that currently gives confidence in output quality is scoped to single-chain, single-model outputs; multi-agent interactions would need a materially more complex eval harness (tracing which agent caused which failure) before they could be trusted in production — that investment doesn't exist yet and isn't cheap to build.

**Concrete trigger conditions that would justify revisiting (none met today):**
1. **Autonomous multi-step execution** — e.g. an agent that actually books flights/hotels across multiple third-party APIs, handles failures, and re-plans mid-transaction. This is genuine multi-step tool-use with real state and rollback needs, materially different from today's single-shot generation calls.
2. **Per-market behavioral divergence at real scale** — if expansion beyond India requires substantively different reasoning/behavior per region (not just prompt/language swaps), a per-market agent split could be justified, but only once that expansion is actually funded and scoped.
3. **Verification workload outgrowing deterministic code** — today's OSM/wiki hallucination verification (`chat_refine_chain.py`'s interest→entity→verify→pin pipeline) is deterministic and cheap; if verification logic becomes complex enough that an LLM genuinely outperforms rule-based checks, a dedicated verifier call (not necessarily a full "agent") might be warranted — evaluate via the existing eval harness before shipping, not by assumption.
4. **Proven eval maturity for multi-step traces** — don't adopt multi-agent patterns until the eval infrastructure can attribute failures to a specific step/agent; shipping multi-agent before this exists would regress the diagnostic clarity that caught the RAG payload-index bug (§6a) in the current architecture.

**Bottom line:** treat multi-agent as a deliberate future architecture tied to a specific new capability (autonomous booking, deep per-market specialization) — not a default "more sophisticated = better" upgrade path. Re-run this assessment when any of the four trigger conditions above is concretely on the roadmap, not on a fixed time/MAU schedule.
